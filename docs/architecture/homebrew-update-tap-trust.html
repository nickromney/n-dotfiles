Problem. Tap-trust config leaked across callers, so the interface for each caller included Homebrew migration env.

Solution. One adapter owns the tap-trust policy and callers cross that seam for mutating Homebrew commands.

  • locality: policy in one module
  • leverage: one interface, many callers
  • tests cross the same seam

Problem. Makefile targets had a shallow interface because each carried Homebrew ordering, warnings, and optionality.

Solution. The caller selects a context; the module owns the implementation sequence.

  • locality: ordering in one module
  • leverage: two callers, one seam
  • tests verify context behavior

Problem. Read-only Homebrew calls sit in the audit and debug modules, but the update/tap-trust behavior does not leak through them.

Solution. Defer a Homebrew inventory module until a second caller needs the same read semantics.

  • locality: not yet worth it
  • leverage: currently low
  • one adapter is hypothetical

Top recommendation

Stop the loop for this path.

The productive change was Homebrew bundle policy coverage. After that, the update/tap-trust path has one policy adapter (scripts/brew-with-policy.sh) and one Makefile update module (scripts/brew-update.sh). The July 2026 restructure removed the config-driven update plan along with the YAML installer; the policy and update adapters survived unchanged. Remaining raw Homebrew reads are speculative until a real second adapter appears.
